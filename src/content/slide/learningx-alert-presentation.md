## 1. 앱 소개

LearningX Alert는 LearningX를 자주 확인하지 못해 생기는 불편을 줄이기 위한 앱입니다.

- 과제, 퀴즈, 토론, 페이지 마감 확인
- 최근 공지 확인
- 과제별 채점 결과 확인
- 마감, 새 공지, 성적 업데이트 로컬 알림

<!--
발표 메모:
LearningX는 학교 수업에서 거의 매일 확인해야 하는 서비스입니다.
하지만 과제 마감, 공지, 성적 업데이트가 각각 흩어져 있어서 놓치기 쉽습니다.
그래서 이 앱의 핵심 목표는 "학생이 놓치면 안 되는 학습 정보를 모바일에서 빠르게 확인하고 알림으로 받을 수 있게 하자"였습니다.
-->

---

## 해결하고 싶었던 문제

- 마감이 가까운 과제를 매번 직접 확인해야 함
- 공지가 올라와도 바로 알아차리기 어려움
- 채점 결과가 업데이트됐는지 확인하려면 다시 들어가야 함
- 네트워크가 불안정하면 이전 정보를 다시 보기 어려움

<!--
발표 메모:
단순히 Canvas API를 호출해서 목록을 보여주는 앱이 아니라, 실제 학생 입장에서 자주 겪는 불편을 줄이고 싶었습니다.
특히 마감 누락 방지, 새 공지 확인, 성적 업데이트 확인, 그리고 오프라인에서도 마지막 데이터를 볼 수 있는 흐름에 집중했습니다.
-->

---

## 2. 구현 기능: 로그인과 동기화

- SSU Canvas WebView 로그인
- Canvas access token 자동 생성 및 안전 저장
- 앱 실행, 새로고침, 앱 재개 시 데이터 동기화
- 실패한 섹션은 캐시 데이터로 대체

<!--
발표 메모:
첫 번째 핵심 기능은 로그인과 동기화입니다.
사용자는 앱 안의 WebView에서 Canvas 로그인을 진행하고, 로그인 후 access token을 생성해 안전 저장소에 보관합니다.
이 토큰으로 Canvas API를 호출하며, 데이터 동기화 중 일부 API가 실패하더라도 전체 화면이 죽지 않고 기존 캐시를 보여주도록 설계했습니다.
-->

---

## 2. 구현 기능: 마감 대시보드

- 앞으로 60일 이내 학습 마감 조회
- 완료된 항목과 숨김 처리된 항목 제외
- 오늘, 내일, 7일 이내, 이후, 마감 지남 그룹화
- D-day와 Canvas 원문 바로가기 제공

<!--
발표 메모:
마감 탭은 이 앱의 기본 기능입니다.
Canvas Planner API에서 과제, 퀴즈, 토론, 페이지 항목을 가져오고, 사용자가 완료했거나 숨긴 항목은 제외합니다.
목록을 단순히 날짜순으로만 보여주지 않고, 오늘과 내일처럼 사용자가 바로 판단할 수 있는 그룹으로 나눴습니다.
-->

---

## 2. 구현 기능: 공지와 성적

- 최근 30일 공지 조회
- 과목별 공지 필터
- 새 공지 배지와 알림
- 최근 120일 과제별 채점 결과 조회
- 점수, 등급, 제출 상태, 지각 여부 표시

<!--
발표 메모:
이번 확장에서 공지와 성적 탭을 추가했습니다.
공지는 과목별 최근 공지를 모아서 보여주고, 새로 들어온 공지는 배지와 알림으로 알려줍니다.
성적은 과목 총점이 아니라 과제별 채점 결과에 집중했습니다.
학생 입장에서는 "어떤 과제가 채점됐는지"가 더 즉각적인 정보라고 판단했습니다.
-->

## 2. 구현 기능: 알림과 설정

- 마감 7일, 24시간, 3시간, 1시간, 15분 전 알림
- 새 공지 알림
- 성적 업데이트 알림
- 알림 권한 상태와 예약 개수 표시
- 테스트 알림 발송

<!--
발표 메모:
알림은 사용자가 놓치지 않게 만드는 핵심 장치입니다.
마감 알림은 여러 프리셋을 두어 사용자가 필요한 시점만 켤 수 있게 했습니다.
또 새 공지나 성적 변경은 이전 캐시와 새 동기화 결과를 비교해 변화가 있을 때만 알려주도록 했습니다.
-->

## 3. 주요 기술 적용

- Flutter + Material 3
- Riverpod 기반 상태 관리
- GoRouter 기반 화면 전환
- Dio 기반 Canvas API 클라이언트
- Secure Storage에 access token 저장
- SharedPreferences 기반 로컬 캐시
- flutter_local_notifications 기반 로컬 알림

<!--
발표 메모:
기술적으로는 Flutter 앱과 별도 API 패키지를 모노레포로 분리했습니다.
앱은 Riverpod으로 상태를 관리하고, Canvas 통신은 learningx_api 패키지에 모았습니다.
토큰은 Secure Storage에 저장하고, 일정·공지·성적 데이터는 SharedPreferences에 캐시했습니다.
알림은 서버 없이 기기 로컬 알림으로 구현했습니다.
-->

- API 패키지와 앱 UI 분리
- 섹션별 부분 실패 처리
- 캐시 기반 오프라인 fallback
- 이전 데이터와 새 데이터 diff 계산
- 테스트 가능한 모델 변환 로직

<!--
발표 메모:

예를 들어 공지 API가 한 과목에서 실패해도 마감과 성적은 계속 보여줄 수 있어야 합니다.
그래서 대시보드 상태에 섹션별 오류를 담고, 성공한 데이터는 갱신하고 실패한 데이터는 캐시를 유지하는 방식으로 설계했습니다.
-->

## 4. 데모 시연 흐름

1. 로그인 화면에서 SSU Canvas 로그인
2. 마감 탭에서 오늘/내일/7일 이내 항목 확인
3. 공지 탭에서 과목 필터와 신규 공지 확인
4. 성적 탭에서 채점된 과제와 점수 확인
5. 설정 탭에서 알림 프리셋과 테스트 알림 확인

<!--
발표 메모:
데모에서는 먼저 로그인 흐름을 보여드리고, 홈 화면이 네 개의 탭으로 구성된 것을 보여드리겠습니다.
마감 탭에서는 D-day와 그룹화를, 공지 탭에서는 과목 필터와 새 공지 표시를, 성적 탭에서는 과제별 점수와 제출 상태를 보여드립니다.
마지막으로 설정 탭에서 알림을 조정하고 테스트 알림을 보내는 장면까지 시연하면 됩니다.
-->

## 데모에서 강조할 포인트

- 앱을 열자마자 중요한 정보가 탭별로 정리됨
- Canvas 원문으로 바로 이동 가능
- 네트워크 실패가 있어도 마지막 캐시 데이터 유지
- 알림 설정이 사용자가 이해하기 쉬운 단위로 제공됨

<!--
발표 메모:
데모에서 단순히 "기능이 많다"를 보여주기보다, 실제 사용자가 편해지는 지점을 강조하면 좋습니다.
예를 들어 앱을 열었을 때 마감, 공지, 성적을 각각 찾기 쉽고, 필요하면 Canvas 원문으로 바로 이동할 수 있다는 점입니다.
-->

## 5. 개발하면서 느낀 점

- API 연동보다 실패 상황 설계가 더 중요했음
- 모바일 앱은 "보여주기"보다 "놓치지 않게 하기"가 핵심
- 캐시와 알림은 작은 앱도 실제 서비스처럼 느껴지게 만듦
- 커밋을 나누고 테스트를 붙이면 확장할 때 부담이 줄어듦

<!--
발표 메모:
개발하면서 느낀 점은, API를 호출해서 데이터를 가져오는 것 자체보다 실패했을 때 어떻게 사용자 경험을 유지할지가 더 어렵고 중요하다는 점이었습니다.
또 모바일 앱은 화면이 예쁜 것도 중요하지만, 사용자가 꼭 봐야 하는 정보를 놓치지 않게 하는 것이 더 큰 가치라고 느꼈습니다.
마지막으로 기능이 커질수록 커밋을 잘 나누고 테스트를 붙이는 습관이 확장성을 크게 높인다는 점도 배웠습니다.
-->
